# BI201 BLE transport

The Glead ZYM-BI201 (TI CC2540) is connected directly to the F411, not the
F103:

| Signal | F411 pin | BI201 function |
| --- | --- | --- |
| F411 TX | PB6 / USART1 TX | UART RX |
| F411 RX | PA10 / USART1 RX | UART TX |
| Module control | PB13 | enable/reset control (exact BI201 pad not traced) |

The stock firmware time-multiplexed USART1 between debug output and the
BI201. Rewair instead dedicates USART1 to the BI201 and compiles with
`WICED_DISABLE_STDIO`; the old `rewair_console` source is not part of the
application. Runtime BLE diagnostics live in the SWD-readable
`rewair_ble_diag` structure. Literal SWO is not available during normal
operation because PB3 is the external SPI-flash clock.

## Bench-proven module interface

The module uses 115200 8N1. Rewair starts it before Wi-Fi initialization:

1. Drive PB13 low for 500 ms, then high. Stock used 50 ms, but 500 ms also
   clears stale GATT/CCCD state after abrupt development disconnects.
2. Wait 1 second.
3. Send `AT#RS\r\n`.
4. Send `AT#WM=0\r\n` to enter transparent mode.

Read-only queries on the tested module returned:

```text
MN=AWAIR-001956
SW=Software Revision
WM=1
FL=1
```

The module advertises this custom service:

```text
2f2dfff0-2e85-649d-3545-3586428f5da3
```

The transparent UART data path is asymmetric at GATT level:

| Direction | Characteristic | Operation |
| --- | --- | --- |
| F411 to BLE central | `2f2dfff4-2e85-649d-3545-3586428f5da3` | notify |
| BLE central to F411 | `2f2dfff5-2e85-649d-3545-3586428f5da3` | write with response |

The negotiated ATT MTU is 23, giving 20 bytes per GATT operation. GATT
notification and write boundaries are arbitrary stream chunks, not Rewair
message boundaries.

## Rewair BLE protocol

`rewair_ble_proto.c` defines protocol version 1. Each message fragment is a
binary record, followed by CRC32, COBS-encoded, and terminated by `0x00`:

| Field | Size | Notes |
| --- | ---: | --- |
| version | 1 | `1` |
| type | 1 | request `1`, response `2`, event `3`, ACK `4` |
| flags | 1 | `FIRST=0x01`, `MORE=0x02` |
| operation | 1 | stable API operation ID |
| request ID | 2 | little-endian, chosen by the central |
| sequence | 2 | little-endian fragment number, starting at zero |
| status | 2 | HTTP-compatible result status; zero in requests/ACKs |
| payload length | 2 | little-endian, at most 32 bytes |
| payload | 0-32 | UTF-8 JSON request/response bytes |
| CRC32 | 4 | little-endian CRC of the header and payload |

COBS plus the zero delimiter lets either side resynchronize after a partial
notification, UART noise, or a reset. A complete request body may be up to
1024 bytes; a complete response may be up to the API worker's 2048-byte
buffer.

The CC2540 bridge has no UART flow control and loses larger F411-to-BLE
bursts. Rewair therefore uses 32-byte logical payload fragments and
stop-and-wait response flow control:

1. The F411 sends one response fragment.
2. If `MORE` is set, the central ACKs that request ID, operation, and
   sequence.
3. The F411 waits 150 ms for the BI201's half-duplex turnaround, then sends
   the next fragment. An ACK times out after 5 seconds.

The F411 also waits 200 ms before the first response to let the completed
BLE-to-UART request drain. These timings and the 32-byte payload were verified
on the physical module; they are not just conservative desktop defaults.

## Transport-neutral API

`rewair_api.c` owns the shared operation handlers and JSON response shapes.
HTTP routes are adapters over that core, and BLE frames carry the same
operations and HTTP-compatible status codes.

| ID | Operation | HTTP route | BLE v1 status |
| ---: | --- | --- | --- |
| 1 | capabilities | `GET /api/capabilities` | enabled |
| 2 | status | `GET /api/status` | enabled |
| 3 | scan | `GET /api/scan` | enabled |
| 4 | networks | `GET /api/networks` | enabled |
| 5 | join | `POST /api/join` | locked |
| 6 | forget | `POST /api/forget` | locked |
| 7 | priority | `POST /api/priority` | locked |
| 8 | settings | `POST /api/settings` | locked |
| 9 | time | `POST /api/time` | locked |
| 10 | display | `POST /api/disp` | locked |
| 11 | reset | `POST /api/reset` | locked |
| 12 | update | `POST /api/update` | locked |

Capabilities currently reports `"mutations":false`. The module allowed
service enumeration, notification subscription, and writes without a pairing
or authorization prompt during bench testing. Mutating operations must remain
locked until Rewair has an authenticated application session (or the BI201 is
proved capable of enforcing an authenticated link). Firmware OTA remains a
Wi-Fi-only operation even after other BLE mutations are enabled.

## Browser UI

The normal Preact UI contains both HTTP and Web Bluetooth transports. On a
page where HTTP status cannot be loaded, it presents an explicit **Connect
over Bluetooth** action, then uses the four read-only operations above and
polls status over BLE. The explicit button also supplies the browser-required
user gesture.

Web Bluetooth requires a supported browser and a secure page context. For
development, Vite on `http://localhost` is suitable:

```sh
cd webui
npm ci
npm run dev
```

For normal use without Wi-Fi, publish `webui/dist/` on any HTTPS static host
(GitHub Pages is sufficient). The Vite build uses relative asset URLs, so it
does not require a Rewair-specific server. `.github/workflows/pages.yml`
publishes the bundle at <https://naorunaoru.github.io/rewair/> after changes
to `webui/` land on `main`; the repository's Pages source must be configured
as GitHub Actions. The copy served directly by the device remains useful for
same-origin HTTP operation, but its plain `http://<device-ip>` origin is not
the deployment path for Web Bluetooth.

## macOS API probe

Bleak uses macOS CoreBluetooth to exercise the same framed API as the browser:

```sh
uv run --with bleak==3.0.1 python tools/bi201/ble_api_probe.py --read-all --quiet-frames
```

This scans for the BI201, connects, subscribes once, and reads capabilities,
status, nearby networks, and saved networks in one session. Omit
`--quiet-frames` to print every response fragment and ACK exchange.

On first use, macOS may ask for Bluetooth permission. If access was denied,
enable it for the host application under System Settings, Privacy & Security,
Bluetooth.

## Standalone hardware probe

`wiced/apps/rewair/bi201_probe` is a small standalone application retained for
module-level investigation. It disables USART1 stdio, wakes the BI201, runs
the AT queries above, enters transparent mode, and publishes a test beacon.
UART replies are kept in a fixed RAM mailbox for SWD readback.

Build, flash only the F411 application slot (preserving bootloader and DCT),
and inspect the mailbox with:

```sh
scripts/build_bi201_probe.zsh
scripts/flash_bi201_probe_probe_rs.zsh
scripts/read_bi201_probe.zsh
```

Restore the normal firmware with:

```sh
scripts/build_local_bridge.zsh
scripts/flash_local_bridge_probe_rs.zsh
```

`tools/bi201/ble_gatt_probe.py` remains available for raw GATT enumeration and
transparent-path testing when debugging below the framed Rewair API.
